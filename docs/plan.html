AI Stock Review Video Harness — Planning 模块修改文档

基于当前 Harness 设计整理 · 2026-05-28

1. 修改目标

当前 Harness 的整体方向合理，但 Planning 阶段需要进一步明确边界： Planner 应该负责数据抓取、清洗、计算和证据整理，而不应该过早输出投资结论。

本次修改目标：

  • 强化 Planner 作为 Evidence Pack Builder 的职责
  • 减少 Planner 对最终投资判断的提前锚定
  • 提升新闻、技术指标、估值数据的可解释性
  • 让后续 Fundamental / Technical / Narrative / Synthesis Agent 拿到更稳定的结构化输入
  • 提升 slides.json 与 narration.txt 的生成一致性

2. Planner 新职责边界

职责 是否属于 Planner 说明
行情数据抓取 是 价格、成交量、市值、52 周区间、估值、财务、分析师数据
技术指标计算 是 RSI、MA20、MA50、布林带、成交量趋势、价格区间位置
新闻初筛 是 按 ticker、公司名、主题关键词做相关性过滤
新闻分类 新增 earnings / product / regulation / analyst / macro / partnership / insider
信号计算 是 只输出 computed signals，不输出最终投资结论
最终买入 / 卖出判断 否 交给 Synthesis Agent 完成
目标价解释 否 Planner 只提供 analyst target 原始数据
股票故事线定性 否 交给 Narrative Agent 完成
幻灯片大纲规划 是 根据 duration 生成结构，但不直接生成内容观点

3. 修改后的 Planner 数据流

Stage 1: Planner

Input:
  ticker / duration / language / market / style

Process:
  1. Fetch market snapshot
  2. Fetch price history
  3. Calculate technical indicators
  4. Fetch valuation and financial data
  5. Fetch analyst consensus
  6. Fetch weekly news
  7. Filter and classify news
  8. Generate computed signals
  9. Generate slide outline
  10. Export plan.json

Output:
  plan.json = structured evidence package
核心原则：Planner 输出“证据包”，不是“投资结论”。

5. 关键修改点

5.1 narrative_signals 改为 computed_signals

原来的 narrative_signals 容易让 Planner 过早进入叙事判断。 建议改名为 computed_signals，强调这是由数据计算出的辅助信号，而不是最终观点。 

Before:
"narrative_signals": {
  "sentiment_score": 72,
  "sentiment_label": "偏强",
  "risk_level": "medium"
}

After:
"computed_signals": {
  "technical_sentiment_score": 72,
  "momentum_state": "strong",
  "valuation_pressure": "high",
  "signal_basis": [
    "Price above MA20",
    "RSI neutral",
    "Analyst upside positive"
  ]
}

5.2 weekly_news 改为 news_evidence_pack

新闻不应该只是标题列表，而应该是后续 Narrative Agent 的证据输入。 因此需要增加分类、相关性说明、匹配关键词和 URL。 

"news_evidence_pack": {
  "lookback_days": 7,
  "items": [
    {
      "title": "...",
      "publisher": "...",
      "published_at": "...",
      "url": "...",
      "relevance_score": 0.86,
      "category": "product",
      "matched_keywords": ["AI", "GPU", "NVDA"],
      "why_relevant": "This news may affect AI infrastructure demand expectations."
    }
  ]
}

5.3 slide_outline 增加 goal 和 required_inputs

原本只用 type 和 approx_seconds 容易导致每页内容重复。 增加 goal 和 required_inputs 后，每一页的生成目标更明确。 

{
  "type": "price_action",
  "approx_seconds": 70,
  "goal": "Explain recent price behavior using price history and technical indicators.",
  "required_inputs": ["price_history", "technical_indicators"]
}

6. 对 Generator 的影响

Agent 输入变化 预期收益
Fundamental Agent 使用 financial_snapshot / valuation_snapshot / analyst_snapshot 减少无关市场噪音，聚焦估值与业绩质量
Technical Agent 使用 price_history / technical_indicators 技术分析更稳定，不依赖 LLM 自行推导指标
Narrative Agent 使用 news_evidence_pack / computed_signals 叙事判断更有证据，降低新闻幻觉
Synthesis Agent 使用三份 Agent 输出 + computed_signals 最终结论由 Synthesis 统一生成，避免 Planner 提前定调
Slides Generator 使用 slide_outline.goal / required_inputs 每页内容更清晰，减少重复表达

7. 推荐后的 Planning 代码模块结构

harness/
├── planner.py
├── planning/
│   ├── __init__.py
│   ├── market_fetcher.py       # yfinance 数据抓取
│   ├── technicals.py           # RSI / MA / Bollinger / volume trend
│   ├── news_filter.py          # 新闻抓取、过滤、分类
│   ├── signal_builder.py       # computed_signals 构建
│   ├── outline_builder.py      # slide_outline 构建
│   └── schema.py               # plan.json dataclass / pydantic schema

如果当前项目还处于 V1，可以先保留单文件 planner.py。 等逻辑稳定后，再拆成 planning 子模块。

8. 第一阶段验收标准

输入：
python run.py --ticker NVDA --duration 9 --skip-eval

期望：
✓ output/plan.json 正常生成
✓ market_snapshot 字段完整
✓ technical_indicators 字段完整
✓ news_evidence_pack 至少包含 title / publisher / category / why_relevant
✓ computed_signals 不包含直接 buy / sell / hold 操作结论
✓ slide_outline 每一页包含 type / approx_seconds / goal / required_inputs
✓ Generator 可以基于新 plan.json 正常生成 analysis / narration / slides

9. 总结

修改后的 Planning 阶段定位为： 

Ticker Input
   ↓
Structured Market Evidence
   ↓
Computed Signals
   ↓
Slide Intent Outline
   ↓
Agent-ready plan.json

这样可以让整个 Harness 更接近产品化流水线： Planner 保证数据稳定，Agent 负责分析判断，Renderer 负责确定性输出。

最重要的原则：不要让 Planner 成为“半个分析师”。Planner 应该是数据工程层，而不是投资观点层。
